并发编程的艺术读书记录

第一章

减少上下文切换的方法：

  • 无锁并发编程：多线程竞争锁的时候，会引起上下文切换，因此可以将数据的ID值按Hash算法分段取模，使用不同的线程去处理不同段的数据。
  • CAS算法：也避免了使用锁。
  • 使用最少的线程：避免创建不必要的线程。
  • 协程：在单线程里实现多任务的调度。

避免死锁的常见办法：

  • 避免一个线程同时获取多个锁。
  • 避免一个线程在锁内同时占用多个资源。尽量保证一个锁只占用一个资源。
  • 尝试使用定时锁，使用lock.tryLock(timeout) 来替代使用内部锁机制。
  • 对于数据库锁，加锁和解锁必须在一个数据库连接里，否则会出现解锁失败。

第二章

volatile的定义和实现原理

定义：

  • Java语言允许线程访问共享变量，为了确保共享变量能准确和一致的被更行，线程应该确保通过排他锁获取这个变量。
  • Java中一个字段被声明为volatile，其内存模型保证所有线程看到的这个变量的值是一致的。

cpu术语的了解：

  • 内存屏障：是一组处理器指令，用于实现对内存操作的顺序限制。
  • 缓冲行：缓存中可以分配的最小存储单位，处理器填写缓存线时会加载整个缓存线，需要使用多个主内存读周期
  • 原子操作：不可中断的一个或者一系列操作
  • 缓存行填充：…

实现原理：反编译发现，加上了lock前缀，会发生两件事：

  • Lock前缀指令会引起处理器缓存回写到内存中
  • 一个处理器的缓存回写到内存中会导致其他处理器的缓存无效：缓存无效以后，再此访问就会进行缓存行填充，因此获取到的值时一定的。

volatile的优化（了解）

在缓存行为64字节宽的处理器中，将volatile变量追加到64字节可以提高效率

实现：

被锁的对象会对应一个monitor对象，基于monitorenter和monitorexit指令来进入和退出锁。

对象头：

synchornized用的锁时存在对象头中的。

偏向锁：

  • 在大多数情况下，不会存在多线程同时竞争锁的情况，而是由一个线程重复获得锁，因此为了减少获得锁的代价，引入了偏向锁的概念。
  • 当一个线程访问同步块并获得锁的时候，会在对象头和栈帧的锁记录里边存缩锁记录的线程ID，以后该线程进入和退出同步块时不需要进行使用CAS进行加锁和解锁，只需要测试一下对象头的MarkWord中是否存储着该线程的ID。
  • 如果测试成功，则获取锁。如果未成功，则去判断该对象的锁的标记位是否为偏向锁，如果没有设置，则使用CAS竞争锁，如果进行了设置，则尝试使用CAS将该对象的对象头的偏向锁指向该线程。
  • 可以通过JVM参数关闭偏向锁：-XX:UseBiasedLocking=false，那么程序默认会进入轻量级锁状态。

轻量级锁：

  • 线程在执行同步块之前，JVM会在当前线程的栈帧中存储锁记录的空间，并将对象头中的Mark Word 复制到锁记录中，官方称为Displaced Mark Word 。然后线程尝试使用CAS将对象头的Mark Word替换为指向锁记录的指针，如果成功，获得锁，如果不成功，则通过自旋来获取锁。
  • 轻量级锁解锁的时候，会使用原子的CAS操作将Displaced Mark Word替换回对象头，如果成功，就表示没有竞争发生，如果失败，就导致锁膨胀为重量级锁。

原子操作的实现原理

Java原子操作的实现：

通过锁和循环CAS的方法来实现原子操作。

JVM中的CAS操作正是利用了处理器提供的CMPXCHG指令实现的。自旋CAS实现的基本 思路就是循环进行CAS操作直到成功为止。

CAS实现原子操作的三大问题：

  • ABA问题：从1.5开始，JDK的Atomic 包中提供了一个AtomicStampedReference来解决ABA问题（增加了一个stamp和expectStamp的校验），解决方法是先使用compareAndSet方法检查当前引用是否等于预期引用，并且检查stamp和expectSatmp是否相等，如果全部相等 ，才以原子的方式将该引用和标志的值设置为给定的更新值。
  • 循环时间长开销大：
  • 只能保证一个共享变量的原子操作：对于多个变量同时操作时，循环CAS就无法保证原子操作，这个时候就要用到锁。还有一种办法就是将两个变量合并为一个变量进行原子操作，这种情况根据需求而定。另外在1.5以后，java推出了AtomicReference类来帮助实现对象的原子性，因此我们可以将多个变量封装为一个对象进行原子操作。

使用锁机制实现原子操作：

锁机制保证了只有获得锁的线程才能够操作锁定的内存区域。JVM内部实现了很多中锁的机制，有偏向锁，轻量级锁，互斥锁。除了偏向锁，其他的锁都使用了循环CAS的方法实现，当一个线程想进入锁时使用循环CAS，退出时也使用循环CAS。

第三章

并发编程模型的两个关键性问题

在并发编程中，需要处理两个关键的问题：线程之间如何通信以及线程之间如果同步。

通信是指线程通过何种机制来交换信息。在命令式编程中，线程之间的通信机制主要有两种：共享内存和消息传递。

Java中采用的是共享内存模型，Java线程之间的通信总是隐式的进行，整个通信过程对程序员完全透明。

JMM通过控制主内存与每个线程的本地内存之间的交互，来为Java程序员提供
内存可见性保证。

第四章

线程

  • 线程优先级

    • setPriority(int)，默认5，1~10，不一定生效
    • 频繁阻塞线程（休眠或者I/O操作）建议设置高优先级
    • 偏重计算（需要占用过多CPU时间）建议低优先级

  • 线程状态

    • NEW 初始状态
    • RUNNABLE 运行状态
    • BLOCKED 阻塞状态
    • WAITING 等待状态
    • TIME_WAITING 超时等待状态 不同于WAITING,可以再自定时间返回
    • TERMINATED 终止状态

  • 线程创建之后，调用start()方法开始运行。当线程执行wait()方法之
    后，线程进入等待状态。进入等待状态的线程需要依靠其他线程的通知才能够返回到运行状
    态，而超时等待状态相当于在等待状态的基础上增加了超时限制，也就是超时时间到达时将
    会返回到运行状态。当线程调用同步方法时，在没有获取到锁的情况下，线程将会进入到阻塞 状态。线程在执行Runnable的run()方法之后将会进入到终止状态。

  • 　Java将操作系统中的运行和就绪两个状态合并称为运行状态。阻塞状态是线程 阻塞在进入synchronized关键字修饰的方法或代码块（获取锁）时的状态，但是阻塞在 java.concurrent包中Lock接口的线程状态却是等待状态，因为java.concurrent包中Lock接口对于 阻塞的实现均使用了LockSupport类中的相关方法。

  • Daemon线程

    • Daemon线程是一种支持型线程，因为它主要被用作程序中后台调度以及支持性工作。这 意味着，当一个Java虚拟机中不存在非Daemon线程的时候，Java虚拟机将会退出。可以通过调 用Thread.setDaemon(true)将线程设置为Daemon线程。

    • Daemon属性需要在启动线程之前设置，不能在启动线程之后设置。

    • 在构建Daemon线程时，不能依靠finally块中的内容来确保执行关闭或清理资源
      的逻辑。

    • 主要用于线程之间的数据传输，传输介质是内存。

    • 主要包括以下四种实现：PipedOutputStream、PipedInputStream、 PipedReader和PipedWriter，前两种面向字节，而后两种面向字符。

            }
        }
        finally{

        }
    }
}

  • ThreadLoacl的使用

    • ThreadLocal，即线程变量，是一个以ThreadLoacl对象为键，任意对象为值的存储结构。这个结构被附带在线程上，也就是说一个线程可以根据一个ThreadLoacl对象查询到绑定在这个线程上的值。

    • public class Profiler {
    private static final ThreadLocal<Long> TIME_THREADLOCAL = new ThreadLocal<Long>();
    protected Long initialValue(){
        return System.currentTimeMillis();
    }
    public static final void begin(){
        TIME_THREADLOCAL.set(System.currentTimeMillis());
    }
    public static final long end(){
        return System.currentTimeMillis() - TIME_THREADLOCAL.get();
    }
    public static void main(String[] args) throws InterruptedException {
        Profiler.begin();
        TimeUnit.SECONDS.sleep(1);
        System.out.println("Cost:"+Profiler.end()+"mills");
    }
}

    • 可以被复用在方法调用耗时功能上，方法的入口前执行begin方法，结束后调用end方法。

    • 每个Thread维护着一个ThreadLoaclMap的引用

    • ThreadLocalMap时ThreadLoacl的内部类

    • ThreadLocal创建的副本是存储在自己的ThradeLoaclMap中的

    • ThreadLocalMap的键值为ThreadLocal对象，并且可以有多个，因此保存在map中

    • 在进行get之前，必须先进行set，否则会空指针，也可以初始化一个，但是必须重写initialValue（）方法。

    • ThreadLoacl本身并不存储值，他只是作为一个key来从ThreadLoaclMap中获取值。

    • 内存泄露问题：ThreadLocal为弱引用，为null时会被清理，而value还在，因此会导致内存泄露。处理办法是ThreadLocal使用完毕后将其remove掉。

  • 使用场景：调用一个方法时等待一段时间，一段时间后，如果能得到结果，就将结果返回，否则直接返回默认结果。

第五章

Lock

  • java SE5 以后出现的，提供了与synchronized关键字类似的同步功能，只不过需要显式的获取和释放锁。
  • 尽量在finally块中释放锁，目的是在获取到锁以后保证会被释放掉。
  • 不要将获取锁卸载try块中，防止获取锁时发生了异常抛出的同时，也会导致锁无故释放。

区别于synchronized 的几个重要特性：

  • 尝试非阻塞的获取锁：当前线程尝试获取锁，如果这一刻锁没有被其他线程获取到，则成功获取锁。
  • 能被中断的获取锁：与synchronized不同的是，获取到锁的线程能够响应中断，当获取到锁的进程被中断时，中断异常会被抛出，同时锁会被释放。
  • 超时获取锁：在指定的截止时间之前获取锁，如果截止时间之前仍然无法获取锁，则返回。
