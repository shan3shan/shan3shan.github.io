3.类的初始化：

在初始化阶段，Java虚拟机会执行类的初始化语句，为类的静态变量赋予初始值。

两种初始化方式：

  • 声明时初始化
  • 在静态代码块中初始化

静态变量的声明语句，以及静态代码块都被看做类的初始化语句，Java虚拟机会按照初始化语句在类文件中的先后顺序来依次执行它们。

步骤：

  • 假如这个类还没有被加载和连接，就先加载和连接。
  • 加入类存在直接父类，并且还没有被初始化，那就先初始化父类。
  • 加入类中存在初始化语句，那就依次执行这些初始化语句。

类的初始化时机：

  • 主动使用七种情况。

  • 当Java虚拟机初始化一个类时，要求它的所有父类都已经被初始化，但是这条规则并不适用于接口。在初始化一个类或者一个接口时，并不会先初始化他们实现的接口。只有当程序首次使用特定接口的静态变量时，才会导致该接口的初始化。

  • 只有当程序访问的静态变量或静态方法确实在当前类或当前接口中定义时，才可以认为时主动使用。

  • 调用ClassLoader类的loadClass方法加载一个类，并不是对类的主动使用，不会导致类的初始化。

类加载器：

类加载器用来把类加载到虚拟机中。类的加载过程采用双亲委托机制，能更好的保证Java平台的安全。在此机制中，除了Java虚拟机自带的根类加载器以外，其余的类加载器都有且只有一个父加载器。当Java程序请求加载器loader加载一个类时，loader首先委托自己的父加载器去加载这个类，若父加载器能加载，则由父加载器完成记载任务，否则才由加载器loader本身去加载该类。

两种类型的类加载器：

  • Java虚拟机自带的加载器

    • 根类加载器（BootStrap）

      无父加载器，负责加载虚拟机核心类库，依赖底层操作系统，并没有继承java.lang.ClassLoader类

    • 扩展类加载器（Extension）

      父记载器为根类加载器，从系统属性所指定的目录中去加载类库，或者从JDK的安装目录的子目录下加载类库。纯Java类，是ava.lang.ClassLoader的子类。

    • 系统类加载器（System）

      父加载器为扩展类加载器，从环境变量classpath或者系统属性java.class.path所指定的目录中加载，是用户自定义的类加载器的默认父加载器。纯Java类，是ava.lang.ClassLoader的子类。

  • 用户自定义的加载器

    • 一定是java.lang.ClassLoader的子类
    • 用户可以定制类的加载方式

类加载器并不需要等到某个类被“首次使用”时才加载

JVM允许类加载器在预料某个类将要被使用时就预先加载它，如果在预先加载的过程中遇到了.class文件缺失或者存在错误，类加载器必须在程序首次主动使用该类时才报告错误，如果一直没有被主动使用，那么类加载器就不会报告错误。

双亲委托机制：

在双亲委托机制中，各个加载器按照父子关系形成了树形结构，除了根类加载器之外，其余的类加载器都有且只有一个父加载器。

各个类加载器的默认加载路径：

类加载器的双亲委托模式的好处：

  • 可以确保Java核心库的安全：所有的Java应用都至少会引用java.lang.Object类，也就是在运行期这个类会被加载，如果这个加载过程是由Java应用自己提供的加载器所完成，那么就有可能会存在多个版本的Object类，而且这些类之间还是不兼容的，相互不可见的（命名空间可能不同），借助于双亲委托机制，Java核心库中的类加载工作都是由启动类加载器来统一完成，从而确保Java应用所使用的都是同样的核心类库。相互兼容。
  • 可以确保Java核心类库所提供的类不会被自定义的类所替代。
  • 不同的类加载器可以为相同名称的类创建额外的命名空间。相同名称的类可以并存在JAva虚拟机中，只需要不同的类加载器。不同类加载器所加载的类之间是不兼容的，这就相当于Java虚拟机内部创建了一个又一个相互隔离的Java类空间，这类技术在很多框架中都有实际应用。

ConTextClassLoader 线程上下文类加载器

认识：

  • 当前类加载器（Current Classloader）,每个类都会尝试用自己的类加载器（加载自身）去加载他所用到的其他类，如果ClassX引用了ClassY,那么ClassX的加载器就会去加载ClassY（前提是ClassX未被加载）

  • 线程上下文类加载器 ContextClassLoader

    • Thread中的getContextClassLoader（）于setContextClassLoader（ClassLoader cl）分别用来获取和设置上下文类加载器。

    • 如果没有设置的话，线程将继承其父线程的上下文类加载器。Java应用运行时的初始线程的上下文类加载器时系统类加载器。

      作用：

    • 父ClassLoader可以使用当前线程Thread.currentThread().getContextLoader()所指定的classloader加载的类。这就改变了了父ClassLoader不能使用子ClassLoader或是其他没有直接父子关系的ClassLoader加载的类，即改变了双亲委托模型。

    • 在双亲委托模型来说，类加载时由下至上的，但是对于SPI来说，有些接口是Java核心库提供的，而Java核心库是由启动类加载器来加载器的，而这些接口的实现是来源于不同的Jar包（厂商提供），这就无法满足。而通过给当前线程设置上下文类加载器，就可以设置上下文类加载器来实现对于接口实现类的加载。

    • 一般使用模式：

      • 获取：ClassLoader classLoader = Thread.currentThread().getContextClassLoader();

      • 使用：try{

        Thread.currentThread().setContextClassLoader(目标类加载器)

        }

      • 还原：finally{

        Thread.currentThread().setContextClassLoader(classLoader )

        }

元空间：（由元空间虚拟机处理）

在以前，类几乎是静态的，很少被卸载和回收，因此类可以被看成“永久代”，并且类是由JVM实现的，不由程序来创建。因此可以看作非堆内存。

在Java8中，取消了永久代，代替其为一个与堆不相连的本地内存区域，也叫元空间。

理想情况下，元空间的大小是不受限制的，可以扩容到本地可用内存的最大值，但是我们还是要对其进行设定大小，否则一旦发生泄露，会占用大量本地内存，这种情况可能更糟糕。-XX:MaxMetaspaceSize ,默认为21MB，可以自动扩容（Full GC调用后）。

垃圾回收

垃圾判断算法

  • 引用计数算法：

    • 给对象添加一个引用计数器，当有一个地方引用时+1，引用失效时-1.任何时刻，计数器为0的对象就是不可能再被使用的。

    • 引用计数算法无法解决对象循环引用的问题。（也就是两个或多个对象互相引用。其实已经不可能再被使用了，但是计数器不为0）

  • 根搜索算法：java中使用的

    • 通过一些列的成为“GCRoots”的点作为起始，进行向下搜索，当一个对象到GC Roots没有任何引用链，这证明不可达。
    • 再VM栈（帧中的本地变量）中的引用
    • 方法区中的静态引用
    • JNI（Native方法）中的引用

方法区垃圾回收：

  • 主要回收两部分内容：废弃常量与无用类
  • 类回收需要满足如下3个条件
    • 该类的所有实例都被回收，JVM中不存在该Class的任何实例
    • 加载该类的ClassLoader已经被GC
    • 该类对应的java.lang.Class对象没有在任何地方被引用。不能通过反射访问该类的方法。

JVM常见GC算法

  • 标记-清除算法

    • 分为标记和清除两个阶段，首先标记出所有需要回收的对象，然后清除所有需要回收的对象。
    • 缺点：
      • 效率：标记清除两个过程效率都不高，需要扫描所有对象
      • 空间：会产生空间碎片，碎片太多可能会导致因为无法找到足够连续内存而提前触发另一次的垃圾收集动作。

  • 标记-整理算法

    • 标记步骤完后，先将所有存活的对象一端移动，然后直接清理掉边界以外的内存。
    • 没有内存碎片
    • 花费时间比标记清除更多。

  • 复制算法

    • 将可用的内存划分为两块，每次只使用其中的一块，当半区的内存用完之后，仅将还存活的对象复制到另外一块上面，然后把原来整块内存空间一次性清理掉。只需要回收半区对象，也不会产生碎片。

    • 缺点：

      • 代价是等价于将内存缩小为原来的一半。
      • 对象存货量高的时候效率有所下降。

    • 现在的虚拟机中都是用了这一中收集算法来回收新生代。

    • 将内存分为一块较大的eden空间，和两块较少的survivor空间，每次使用eden和其中一块survivor，当回收时将eden和survivor空间上的活对象一次性拷贝到另一块survivor空间上，然后清理掉Eden和用过的survivor

    • Hotspot虚拟机默认的eden和suivivor的大小比例时8：1，也就是每次有10%的内存是浪费的。

  • 分代算法

    • 将Java堆分作新生代和老年代。这样可以对不同的年代的不同特点采用最适当的收集算法。例如在新生代采用复制算法，在老年待采用标记清除或者标记整理算法。

    • 年轻代：

      • 新生成的对象都放在年轻代，年轻代用复制算法进行GC（年轻代对象声明周期较短）
      • 年轻代分为三个区。一个eden，两个survivor（可以通过参数设置survivor的个数）。对象在eden区中生成，当eden区满时，还存活的对象将被复制到一个survivor区，当这个survivor区满时，此区的存活对象将被复制到另一个survivor区，当第二个survivor区也满了的时候，从第一个survivor区复制过来的并且此时还存活的对象，将被复制到老年代，两个survivor是完全对称， 轮流替换的。
      • 比例是8：1：1，可以根据GC log 的信息调整大小的比例

    • 老年代

      • 存放了经过一次或多次GC还存在的对象。
      • 一般采用标记-清除或者标记整理算法。
      • 有多种垃圾回收器可以选择，每种垃圾收集器可以看作一个GC算法的具体实现。

    内存回收：

    • Hotspot将引用分为四种：Strong（强），Soft（软），Weak（弱），Phantom（虚）

      • Object o = new Object（）为Strong
      • 其余都继承Reference

    • 内存回收时对Reference的特殊处理（Full gc）

      • Soft：内存不够会被GC，长期不用也会被GC
      • Weak：一定会被GC
      • Phantom

    GC的时机：

    • Scanenge GC （Minor GC）

      • 触发时机：新对象生成时，Eden满了
      • 理论上Eden区大多数对象会被回收，复制算法的执行效率会很高，Scavenge GC时间比较短

    • Full GC

      • 对整个JVM进行回收，包括Young，Old，Perm
      • 主要触发时机
        • Old满了
        • Perm满了
        • system.gc（）
      • 效率很低，尽量减少（避免STW）

    垃圾收集器的并行和并发：

    • 并行：多个收集器的线程同时工作，但是用户线程处于等待状态。

    • 并发：只收集齐在工作的同时，可以允许用户线程工作

      • 并发不代表解决了GC停顿的问题，再关键的步骤还是要停顿，比如收集器在标记垃圾的时候。

    • Sreial单线程收集器：收集时会暂停所有工作。STW。使用复制收集算法，虚拟机运行在Client模式时的默认新生代收集器。

    • ParNew收集器：

      • Sreial收集器的多线程版本
      • 收集时会暂停工作，产生STW
      • 新生代复制。
      • 只有在多CPU的环境下，才会比Serial收集器效率高。
      • 可以通过-XX：ParallelGCThreads来控制线程数的多少，需要结合具体CPU的个数。

    • Parallel Scavenge收集器：

      • 也使用复制算法
      • 是以吞吐量最大化（即GC时间占总运行时间最小）为目标的收集器实现，允许较长时间的STW换取总吞吐量的最大化。

    • Serial Old收集器：

      • 单线程，使用标记整理算法，是老年代的收集器。

    • Parallel Old收集器：

      • 老年代版本吞吐量优先收集器，使用多线程和标记整理算法。
      • 注重吞吐量。

    • CMS收集器：（并行标记清除算法）

      • 以最短停顿时间为目标
      • 追求最短停顿时间，适合Web应用
      • 只针对老年代，一般与ParNew收集器结合使用
      • 和用户线程并发工作（尽量并发，标记垃圾会停顿）
      • 只有在多CPU才有意义
      • 使用-XX：+UseConcMarkSweepGC打开
      • 以牺牲CPU资源的代价来减少用户线程的停顿。
      • 使用标记清除算法，会带来碎片问题，频繁会导致Full GC

      内存泄漏经典原因：

      • 对象定义在错误的范围
        • 该定义在为局部变量去额定义为成员变量
      • 异常处理不当
        • 在处理异常时没有考虑到资源是否一定会被关闭。JDBC连接时。
      • 集合数据管理不当
        • 使用基于数组的数据结构时，尽量减少resize
        • 减少resize可以避免没有必要的array copying
