数据库相关问题

  • 聚集索引和非聚集索引

    聚集索引：索引树的叶子节点存储的是数据（InnoDB引擎），也就是数据的逻辑地址和物理地址都是连续的，因此查询的速度相对来说会比较快。由于这种特殊的要求，所以一张表只能有一个聚集索引。

    非聚集索引：索引树的叶子节点存储的是数据的存储地址（ MyISAM ），因此数据的逻辑地址和物理地址一般是不相同的。

在foreach中使用add/remove 方法出错

在阿里开发手册上有一个强制规定，不要再foreach循环里进行元素的add/remove操作，remove元素要使用Iterator 方式，如果并发操作，要对Iterator对象加锁。

原因：我们在使用foreach的时候，其实时使用itertor进行遍历的，在生成itertor的时候，会保存一个expectedModCount参数，这个表示List中修改元素的次数。如果在遍历过程中删除元素，List中的modCount就会变化，如果这个modCount和expectModCount不一致，就会抛出异常。

通过阅读源码，确定modCount这个值是在创建一个list的时候初始化的值，初始值是0，每次list发生改变都会自增。expectModCount则是在初始化一个迭代器Itr时候定义的值，和modCount相等。我们知道foreach是通过迭代器实现的，所以会初始化一个迭代器，在遍历时如果有添加或者删除，list的modCount就会发生改变，此时就不再和expectModCount相等了，因此会出现ConcurrentModifcationException。此处是java的fail-fast（快速失败）机制。

另附：用上方法删除倒数第二个方法不会出错。原因：我们通过对程序打断点，我们发现产生异常的位置是在next（）方法中，而在Itr方法中，只有haNext（）方法返回true时候才会进入next方法。在Itr中，有两个属性，cursor（当前位置–光标），lastRet最后一个元素下标，在删除倒数第二个元素后，cursor就会指向最后一个元素，此时lastRet就与cursor相等了，不会进入next方法，就不会报错了，但是最关键的问题是，此时list根本没有被遍历完。最后一个元素没有遍历。

在项目中使用List进行对比去重的时候在foreach中使用了remove方法。导致程序报错。最后改为在删除时先在遍历时将要删除的元素放入一个新的集合中，最后调用removeAll方法进行删除。

方法二：

// 迭代器循环遍历
List<String> stringListFor = getStringList();
System.out.println("删之前："+stringListFor);

Iterator<String> iterator = stringListFor.iterator();

while (iterator.hasNext()){
    if(iterator.next().equals("chengxi")){
        iterator.remove();
    }
}
